Introduction to python scripting tools

Introduction

Mistika Workflows have integrated support for Python programming language (Currently Python 3.11). This language provides bidirectional access to all the Workflows data structures, and most GUI actions are also available to python scripts. You can also create Python nodes very easily.

For advanced users It is recommended to learn these tools in order to achieve full automation for complex or repetitive workflows, and to get a more customized experience tailored to your needs.

This introduction chapter is intended for users with at least a basic python understanding (this is not a python course). However no python expertise is needed for basic Python usage, in fact it can be quite the opposite: Mistika Workflows is designed to hide a good part of the complexity of traditional Python programming, thanks to its ability to control flow execution via the node graphs, also offering a dedicated GUI with execution queues and full structured access to common media files and its metadata.  Our ultimate objective is that the users will only need Python programming for very particular workflows not following common market standards, while Mistika Workflows will provide pre-built nodes for everything else. Additionally, SGO offers optional custom programming services on demand, for developing nodes to client specifications.

The python scripting tools are available in the Window->Panels->ScriptEditor.  Normally each python script is stored (and executed) in a Python node, but a python interpreter is also provided.

Note: It is recommended to  prepare one GUI layout for programmers with those tools  and a different one for normal users. (You can do it with Window->Layout->Load / Save)

Note: Mistika Workflows versions before 10.8 version were using Python 2.7, which is not fully compatible with Python 3.xx language that is used since Mistika Workflows 10.8.  If you have a workflow using older Python nodes they will need to be upgraded:  If you got those nodes from the Finder panel then you can simply select them and click on Contextual menu->Code update->Finder to node function to update them to the latest Python version (but logically this only works for official nodes coming with the software and not for custom implementations, which will require to be upgraded by their programmers)

Warning: Python 3.xx syntax is very similar to Python 2.7, but not identical. Some code examples on this manual were originally written in Python 2.7 and may fail on current version. This manual is being reviewed to update the most significant examples to Python 3, but most examples were got from older forums or not maintained by their  authors and may still remain with the old syntax.  However, the differences are minor and the Mistika related  concepts have not changed (and this manual is not intended as a Python beginners tutorial anyway).

IMPORTANT NOTE: The integrated text editor is very basic, and for complex scripts you will prefer to use an external text editor. If you do that, when you want to copy / paste Python code from external documents, if you plan further modifications it is highly recommended to paste it into the Script Editor panel, not directly into the Python code property of the node.

This is because the Code section of the Python node is not really a text editor, just a property container, and it may fail to deal with external formatting, custom indentations  and other hidden codes embedded in the pasted text.  Meanwhile the Script Editor provides full text conversion and it is Python-syntax aware.

Also note that the Code section of a Python node belongs to a node instance, no code is saved anywhere until you do it explicitly, and any change to the code will disappear as soon as you delete that node instance. If you want to keep your changes you need to use the contextual menu -> Save in library to create or update the python node in the workflows library.

 

 

 

Mistika classes

 

Mistika Workflows exports a number of modules to give access to the system functionality.

 

 

This is just an introductory document with few examples,  the full reference guide with all Mistika classes is available in the Mistika python extension library.

 

A first python example: Executing these command lines will create a workflow "my test" with two nodes (a "File" node and a "Prores" node), and it will interconnect both of them.

from Mistika.classes import Cconnector

from Mistika.classes import CbaseItem

w=Mistika.workflows.addWorkflow("my test")

inputNode=w.addNode("File",CbaseItem.NODETYPE_INPUT)

outputNode=w.addNode("ProRes",CbaseItem.NODETYPE_TASK)

inputNode.getFirstConnectorByName("To").link(outputNode.getFirstConnectorByName("VideoIn"))

 

Note that in this example we have imported the CbaseItem class to have access to predefined constants. The equivalent without this constant is possible but it would be less legible, as it would be something like:  

inputNode=w.addNode("File",1)

 

Later you can access that node from other scripts and other nodes:

w=Mistika.workflows.getWorkflow("my test")

 

You can also access the active workflow without the need of knowing its name:

wf=Mistika.workflows.getCurrentWorkflow() (gets the current workflow from the active tab)

 

And now let's get a property of one of its nodes (the color):

n=wf.getNode("File")  

print n.color

 

Interestingly, the python id of a property does not need to match what is shown in the GUI. In this example the property appears as "Color" in the GUI, but its python id is "color" (lowercase). The easiest way to know the python id is to place the pointer on the property name, which in this example will show a tooltip saying "Python id: File.color". (the labels for the GUI are explained later)  

 

 

Python nodes

 

A “Python” node can be used at any part of a workflow. It is capable to execute standard Python code, but also with easy access to all the data structures of Mistika Workflows and useful metadata of the media files (most metadata obtained or created by one node is passed between nodes from the input to the output connectors, so that it is accessible across the whole workflow). And the most important functions of the GUI functionality are also available as python predefined classes and functions.

For example, the code contained in just one Python node can not only do things with its input files and metadata, it can also create new workflows with any type of nodes, then send those workflows to the jobs queue and execute them. Basically it can do similar things as a human user could do in the application but in automated way. This capability provides a powerful automation tool.  

Without python nodes a normal user can still order the mass processing for thousands of files, but this is only practical as long as the processing needs for those files have few variations between them. Meanwhile, using Python nodes is the right way to go beyond those limitations. For example you can use Python to read a complex production spreadsheet and then create hundreds  of simultaneous workflows, all different between them and with each one having its own variations and specialties depending on the sheet content.

You can also use Python to create new nodes and add them to the Workflows node library, so once they are operative they can can be made readily available to normal users with no Python knowledge, and also with control about what properties they can see or not in the properties panel (this is explained in next sections)

Now going into detail:

A Python node need to have at least 3 functions defined (you can load a Tasks->Python node to see an example):

init(self): Code to be executed when creating a new node instance. Return True if no error, or False in case of error. Typically used to create the node properties and connectors.

isReady(self): It must return True if the node has everything that it needs to run, or False if not yet. As a difference to the init procedure this function is being  evaluated pretty often to refresh any change in the workflow or any changes in node parameters. This is typically used to validate if the node has all what it needs (including connectors). The light indicator of each node will also reflect the IsReady state at any moment.

As a result of this, in some cases the isReady function can be executed many times in few seconds to refresh ongoing changes, so you should only include very basic checks on it!. For this reason, If you just want to validate property values when they are changed, a more convenient method is the onPropertyUpdated  (explained later)

process(self): The actual code to execute when running the workflow in the job queue. it must return True if no error, False in case of errors.

The self parameter provides the object id to the node. You will need it when creating properties and connectors, read their values, and adding functionality to the node in general.

Example: (a “hello world” node):

#This node will print a message in the Script Editor console

def init(self):

  return True  

def.isReady(self):

  return True   

def process(self):

  print  “Hello World, my name is ”+self.objectName

  return True   

 

When looking for examples, please note that in the node library all the nodes starting with "py" (pyDoNothing, pyHelloWorld, pyPrintInputs etc) are standard python nodes, so you can see their code as a reference.

A typical way to create new python nodes them is:

- Load a Python node to use it as an example template

- Rename the node (this is important, use a meaningful name for you)

- Change the code

- Save in library. The SaveInLibrary  action (available in the contextual menus of the node and the script editor) will save a file with your current python code under the SGO AppData/shared/workflowsLibrary, which is will make it to appear as a new python node in the nodes library.  

Note: By default SaveInLibrary will place the new nodes in the Tasks category of the library, but you can force a node to appear in any other category  by using the setSupportedTypes node function. However this function is a bit too advanced at this stage of the introduction, but if you want to see a full example check the Outputs->pyPrintPinputs node and take a look to the init function of its code (it has one line executing setSupportedTypes).

 

 

Changing node state and sending messages to the user trough the validation inspector and render log panels.

 

Your node can use these commands to send messages to the GUI for user knowledge, both via validation inspector and render log panels, similar to other nodes do:

 

self.info("msgType","message content" )

self.warning("msgType","message content" )

self.critical("msgType","message content" )

 

Where msgType is just an internal id name for your own control, you can type anything meaningful for you.  

The info/warning/critical property control the icons and colors to be displayed with the message. Logically, to keep the node representation coherent the return code from isReady and process should always return false for the case of "critical" messages and true otherwise (as a good practice those methods should never return false without also sending a critical message to explain the reason to the user). Note that returning false from the isReady procedure will also prevent  workflow execution, which is useful to prevent mistakes and to inform the user that the node is not ready for execution yet.

Note: You can get the current status of a node (and status icon) by reading its colorByStatus property, which is set to True or False depending on the isReady() return value.

 

 

 

Adding new Python nodes to the workflows library and managing their properties in the GUI.

 

Adding a new user node to the nodes library.

You can add new python nodes to the nodes library, and they will appear in the task section of the Finder panel and other places, as if they were any other original nodes coming with the application. To do that, you can place the node code in the Script Editor and use its contextual menu option Save in Library. It will ask a name for it and the new node will appear in the Finder panel. Alternatively,  you can also do it manually:  Just put your python code into a text file with ".py" extension, and place it in this folder:

SGO AppData/shared/workflowsLibrary/nodeName.py

 

This method will load the node the next time you open Mistika Workflows, while using the Save in Library method in the contextual menu of a node works immediately, without the need to reload the application.

Note: That is just the default location for the shared folder. When using multiple computers (or multiple Mistika products) you may prefer to copy the shared folder to a shared storage (hence its name), . The Mistika "shared" folder not only permits to share your nodes library but also other important assets like Mistika the display_filters folder (for the Mistika predefined effects), LUTs, and in general other Mistika files independent from the local system. The Mistika paths structure is explained here.

 

Removing a user node from the nodes library.

To remove a user node, just delete it from the library folder (and restart the application to reload the library): SGO AppData/shared/workflowsLibrary

 

 

 

Testing  python code in the script editor, with autofiller support

 

In a python node the self variable containing the node id. is given to your code, it is not a variable that you need to define. But sometimes you may want to test some code from the node directly in the script editor, or use it with the autofiller to write code.  The command line and the script editor are in the same python context, but they are not connected to any node (self is not defined). To solve this, a simple way is to create a node and then define the self variable, by executing this in the command line:

from Mistika import workflows

w=workflows.getCurrentWorkflow()

self=w.getNode("YourNodeName")

 

In that way,  from this point the autofiller will also work with the self variable in the script editor, and you can execute code from it.

 

 

 

Creating properties and controlling their visibility in the Properties panel:

Your Python nodes can have any number of properties. Some of them will come predefined, but you can create as many as you want in the init procedure. Example:

 

def init(self):

  self.addProperty("myProperty1", "this is my default value")

  return True

And then use it in the rest of your code.  Example:

 

self.myProperty1="hello"

 

The property values will be stored in the node instance until deleting it.

In the previous example a property myProperty1 was created, but it will not be visible to the user yet. You can control which properties appear in the properties panel, and also hide other properties that you don't want normal users to see (like the python Code property for example). To do that, each node have a .xsd definition file in "SGO AppData/shared/config/schemas" , with the same name as your python node and .xsd extension:

SGO AppData/shared/config/schemas/nodeName.xsd

The first time that you use the node a default xsd file will be automatically for it, with no properties visible (only default properties, objectName and color).

Note that the python nodes come with some predefined properties (like the color) which may or may not be visible in the default xsd.

You can edit the .xsd files manually (which is recommended for simple cases, for example by using the xsd of other nodes as a reference), but an specialized tool is also available in the Window->Panels->Properties editor for more complicated cases.

 

 

 

This tool is only for very complicated cases with many properties, so you should not need it at the beginning. So we will only provide a brief introduction: In this tool first you have to select your new node in the Base Item Selector. Then drag and drop properties from the Base item view to the Designer area to make them visible, You can also create groups, drag & drop properties between them etc.  The widgets functionality  follows the Qt standard which is not documented here, but there is plenty of documentation on Internet.  We recommend to use existing nodes as a reference to learn the possibilities. But just remember that anything you do with this tool should match the node properties already in your Python code, otherwise  it will not work or it will produce erratic behavior.

Basic properties are strings and boolean (shown as checkmarks ). But the properties panel can also offer combo boxes to the user to select values from a list. For example, for a property called "time" you could have this line in the xsd file of the node:

<xsd:element name="myTime" type="integerPulldown" sgo:integerPulldownValues="One second$$$1|One minute$$$60|One hour$$$3600" sgo:visibility="0" sgo:sortOrder="10"/>

 

Which will offer the 3 possibilities to the user (One second, One minute, One hour) and will set the property "time" to the corresponding integer ( 1, 60, 3600 ).

 

You can also get the values offered in the combo box from a different property, for cases where value content depends on executing some other code. For example you could define these properties:

 

self.addProperty("bet")

self.addProperty("_championsLeagueFinalists")

_championsLeagueFinalists=["Real Madrid", "Chelsea"]

 

And then have this line in the xsd file:

 

<xsd:element name="bet" type="stringPulldown" sgo:actionType="getDataFromProperty" sgo:actionTypeProperties="_championsLeagueFinalists" sgo:visibility="0" sgo:sortOrder="10"/>

 

The python id of a property does not need to match what is shown in the GUI via the xsd file.  Remember that the easiest way to know the python id of a property is to select the node in the GUI and place the pointer on the property name. This action will show a tooltip with the python id after a second or so.

 

If you need to obfuscate a property (like when requesting a password) you can use the addEncryptedProperty function. Check the Mail task node for an usage example.

 

 

 

Changing property visibility dynamically

 

Once a property is already in the xsd file (and not before), then you can use this method to make it temporally invisible, for example to make it appear depending on the values of other properties.

self.setPropertyVisible("propertyName",False)  

And then set it back to True to make it visible again.

 

 

Reacting to property updates

 

When the value of a user defined property changes (either by a user action or programmatically) the isReady function is invoked. So that function was a traditional place for checking property changes and to react accordingly.

 

However, since recent versions there is a more convenient way to do this, by implementing the function:

 

onPropertyUpdated(self, name)

     

This method is recommended for more legible code, and specially because it is only executed when a property changes, while isReady is invoked much more often on any type of change (even a simple click on the node can be enough to trigger isReady several times..).

 

Note: Please pay attention to remember this point because you will still find the old method on most of the code examples. They were made before this function was available, so it is easy to forget.

 

Example: Whenever the user (or a python command) changes the "country" property of this node, it will force the first character to be uppercase.

 

def init(self):

   self.addProperty("country")

return True

 

def isReady(self):

   #Check here if the node is ready to be processed

return True

 

def onPropertyUpdated(self, name):

if name == "country":

  self.country=self.country.capitalize()

  print "Country changed: self.country: "+self.country

 

def process(self):

  #Do your processing here

return True

 

You can find an usage example in the code of the  Classify By Extension node, where it is used to re-invoke the init function to add and remove connectors on the fly.

 

 

 

Creating persistent variables

 

When deleting a python node all its python variables are destroyed with it. If you need to keep persistent variables accesible across your workflows you can use other methods, for example:

1- Using custom metadata tokens. You can add your own metadata tokens to this file:

SGO AppData/shared/config/NameConventionPlaceHolders.xml

Then you can use pyhton nodes to populate their values, being a good example the NC to metadata node (use its python code as a reference).

This method is recommended when you need to keep the value across a workflow execution.

 

2- Another method is by using environment variables of the operating system:

import os

os.environ["MY_VARIABLE"]="my_value"

 

This method is recommended when you need to keep the value during  a whole Mistika Workflows session. However they only work for simple strings, for more complicated structures that you want to be persistent it is recommended to store them in files.

3- Reading and writing variables to your own metadata files. Required to keep metadata between sessions, even surviving to potential crashes.

 

 

 

Adding drag&drop capability to a property

Python nodes support drag&drop, to drop files or folders from external file browsers. First you need to tell which property will receive the drag&drop actions with setDropToProperty:

 

self.addProperty("myProperty1")

self.setDropToProperty("myProperty1")

 

You can also declare a list of acceptable file specs.:

 

setDropSupportedFileMasks (["*.dpx","*.jpg"])

 

Optionally, you can define a processDroppedFile method to be invoked during drag&drop actions, and it will be passed the dropped path as parameter:

 

setDropSupportedFileMasks (["*_LEFT_*", "*_RIGHT_*" ])

 

def processDroppedFile(self, path)

    #This example gets the drag&drop of one eye and it loads the second eye

    oneEye=str(path)

    self.rightEyeProperty=oneEye.replace("_LEFT_","_RIGHT_")

    self.leftEyeProperty=oneEye.replace("_RIGHT_","_LEFT_")

 

 

 

Adding workflows to the task queue and executing them from within a script.

 

Your python scripts have access to a task global variable to control the task queue and its execution.  For example this script will create a workflow and it will add it to the task queue:

import Mistika

from Mistika import workflows

from Mistika.classes import CbaseItem

w=Mistika.workflows.addWorkflow("MyWorkflow")

tasks.addWorkflow(w)

 

To execute all workflows, you can use this:

tasks.runAllTasks()

 

The above case is just illustrative, but there is a simpler way to do just that in common cases:  if what you want is to execute Mistika workflows,  load a workflows file (.mwf), and execute all its workflows in the queue you just need a simple command line to be executed on your operating system. Examples:

Windows: "PathToWorkflowsExecutble\workflows.exe" -A "C:\Path\To\Workflows\File.mwf"

Linux, Mac: "PathToWorkflowsExecutble/workflows" -A "/Path/To/Workflows/File.mwf"

 

 

Working with universal paths and node connectors

 

The node connectors communicate by sending universal paths. As a difference to a standard file path the universal paths permit to include extra metadata parameters, typically used to pass composite media packs (such as an enumerated file sequence or other composites made of multiple files) and additional metadata.

 

This example creates a Task with one input and one output. On execution, this example just sends the input (if any) to the output. This example does not process the input file.

 

from Mistika.classes import Cconnector

 

def init(self):

   self.addConnector("input",Cconnector.CONNECTOR_TYPE_INPUT,Cconnector.MODE_OPTIONAL)

   self.addConnector("output",Cconnector.CONNECTOR_TYPE_OUTPUT,Cconnector.MODE_OPTIONAL)

   return True

 

def isReady(self):

   return True

 

def process(self):

   input=self.getFirstConnectorByName("input")

   output=self.getFirstConnectorByName("output")

   output.setUniversalPaths(input.getUniversalPaths())

   return True

 

 

Connectors can create universal path objects, work with them and then pass them to outgoing connectors. Example:

p=CuniversalPath(self.getName Convention())

p.setPath("C:/My/Path/to/CarCatalog")

p.setParam("car", "BMW 325i")

out=self.getFirstConnectorByName("output")

out.clearUniversalPaths()       

out.addUniversalPath(p)

 

Example:  During node execution, read incoming universal paths from all connectors:

for c in self.getConnectors():

   for p in c.getUniversalPaths():

        print "Full universal path: ", p.toString()       

        model=str(p.getParam("car",""))

        catalog=str(p.getParam("root",""))

        print "Catalog: ", catalog, "Model: ", model      

 

The nodes are responsible to clear their output values when needed, (typically at the end of the execution). Otherwise the output values are kept until they are replaced in a new execution, which can be dangerous if your node does not replace them (for example when used in bypass mode). You can clear all the outputs by adding this function at the end of the process function:

out.clearUniversalPaths()

 

A very useful node for debugging is the pyPrintInputs. You can connect any number of outputs to the inputs of this node and it will simply print their content in the script editor.

 

Python nodes can have a variable number of connectors (a good example is the pyPrintInputs mentioned above), which can be dynamically added by the user (via drag & drop actions) or by using python commands. This is controlled by the setAcceptConnectors property. The following usage example is from the Classify By Extension node, which admits a variable number of input connectors and will send the input files to different outputs depending on their file type:

from Mistika.Qt import QColor

from Mistika.classes import Cconnector

 

def init(self):

# creating properties

    self.addProperty("errorMode",0)

    self.addProperty("extensions","mov,mxf,mp4,j2k,r3d,rnd")

#creating connectors

    self.addConnector("files",Cconnector.CONNECTOR_TYPE_INPUT,Cconnector.MODE_REQUIRED)

    extList=self.extensions.split(",")

    for ext in extList:

        self.addConnector(ext.strip(),Cconnector.CONNECTOR_TYPE_OUTPUT,Cconnector.MODE_OPTIONAL)

    self.setAcceptConnectors(True,"files")

#configuring the node

    self.bypassSupported=True

    self.color=QColor(0,180,180)

    return True

 

def isReady(self):

    res=True

    if int(self.errorMode)<0 or int(self.errorMode)>3:

        res=self.critical("filesSplitter:isReady",u"Invalid errorMode {}".format(self.errorMode),"errorMode") and res

    return res

 

def process(self):

    #And finally, Do the thing here!

    res=True

    inputs=self.getConnectorsByType(Cconnector.CONNECTOR_TYPE_INPUT)

    outputs=self.getConnectorsByType(Cconnector.CONNECTOR_TYPE_OUTPUT)    

    for c in outputs:

      c.clearUniversalPaths()

    if self.Bypass:

        return True

    for c in inputs:                                 

        for up in c.getUniversalPaths():

            ext=up.getExtension().lower()

            out=self.getFirstConnectorByType(Cconnector.CONNECTOR_TYPE_OUTPUT,ext)            

            if out:

                self.info("filesClassifier:process:info",u"sending {} to {}".format(up.getFileName(),ext),"") 

                out.addUniversalPath(up)

            else:

                func=[None,self.info,self.warning,self.critical][int(self.errorMode)]

                if func:

                    res=func("filesClassifier:process:ext",u"Extension not supported {}".format(ext),"") and res

    return res

 

def  onPropertyUpdated(self,name):

     if  name=="extensions":

        init(self)

 

When your code needs to interconnect nodes with a variable number connectors you can use the dropConnector function, which simulates the user action of dropping a connector from another node (without the need to take care of connector position details).

 

Bypass support

 

Many nodes  support the bypass mode (a little eye icon on the node), and you can also implement it for your python nodes via the self.bypassSupported property (as in the previous example). The typical usage is to let the user to bypass the node for testing purposes temporally, without having to disconnect and reconnect it from the other nodes.

 

In this mode, normally you will want your node to return without any action, but you may want to pass the inputs to the outputs or to implement any custom behavior. It really depends on what your node does in the workflow.

 

If you want to support bypass you need to it explicitly, as follows:

 

- In the init function, set the bypassSupported property as True. That will show the bypass icon in the node.

 

self.bypassSupported=True

 

- In the process function, manage the bypass as preferred. Example:

 

if self.Bypass:

   #do your actions for bypass mode

   return True

 

And make sure the Bypass property appears in the .xsd file of the node (they are stored in the SGO AppData/shared/config/schemas folder), which should have a line like this:

 

<xsd:element name="Bypass" type="xsd:boolean" sgo:visibility="0" sgo:sortOrder="2"/>

Code injection

 

You can inject python code into a text variable or a property and have it executed at evaluation time, by using this syntax:

 

<? Your python code ?>

 

For example, if you want to send an email with a listing of the input files you could put something like this in a property:

 

myInjectedCode="""Hello guys, these are the list of files that I have received:

<?py

for c in self.getConnectors():

   for p in c.getUniversalPaths():

      print p.getFilePath()

?>

 

Best Regards,

 

The Workflows team"""

 

Which can be evaluated at execution time by putting something like this in the process function:

 

 body=self.evaluate(self.myInjectedCode)

 

 

Typical usage is when you want to easily compose text with procedural data, or when you want to inject code into another node (by populating one of its properties and letting that node to evaluate it later)

 

See the mail node example for a typical usage example

External CLI interface

 

Starting Workflows with a .mwf file and executing it automatically ( -A & -r parameters)

 

You can execute workflows by command line. The -A parameter will load a .mwf file, then it will send all its workflows (the ones that are in "ready" state) to the queue and then it will start their execution, similar to what a human operator will do. The interface will remain open so that you can monitor their progress. Example (on windows):

 

workflows -A “C:\MyPath\MyWorfkflows.mwf”

 

For example you can add that command line to the task scheduler of your operating system to restart your workflows automatically if the application crashes or if the session disconnected (login/logout cycles), for systems running unattended sessions. This link provides an example on Windows, but unix systems also have similar procedures.

 

Alternatively, the -r parameter will do the same, but it will force Mistika Workflows to exit after the initial execution.  This prevents multiple instances of Workflows to stay "zombie" after their command line executions if they are not expected to do anything else.

Example (on windows):

workflows -r “C:\MyPath\MyWorfkflows.mwf”

 

Quiet mode with -q

 

The appearance of the interactive GUI is always optional: it will appear by default, but it can be hidden with the  “-q” parameter (quiet mode).

workflows -q -R “Path/To/Python/Script.py”    

 

Starting Workflows and passing a  python script to it with -R

The Mistika Workflows executable can be passed something like “-R MyScript.py” parameter to open the application and then run a Python script automatically. The Mistika python classes provide access to most of the Workflows functionality, so you can use python scripts for additional automation and integration with other applications:

workflows  -R “/Path/To/PythonScript.py”    

Note:  requires forward slashes "/" notation, even on Windows

 

Once the script is executed the Mistika Workflows application will stay running (so the user can  interact with it).  This is the default behavior, but you can change it as follows:

if the application is executed in quiet mode (or if you want it to exit at the end of the script execution) then you need to add a Mistika.workflows.quit() command at the end of the script. Otherwise Mistika Workflows will not realize when the script is finished and needs to exit, so the application will seem to be frozen forever, also  blocking system RAM unnecessarily. A typical external script looks like this:

import Mistika

...script content...

Mistika.workflows.quit()

 

After executing the script, you can get the render log in this file (it will have the same output as if you execute it in interactive mode, with all the info/warning/critical messages triggered by the python node): SGO AppData/Workflows/log/workflowsRender.log

 

Your external script can create and execute a workflow or the whole workflows queue as said:

tasks.runAllTasks()

 

But if you are using a external script then it is important to note that the render tasks will be executed in the background. That means that your script can (and will) continue executing while the render is happening and also after that.  This level of control can be very useful but it is important to not let your script to reach the final Mistika.workflows.quit() command before the render is finished.  For example, you can use the class CrenderTask, which provides an isCompleted method to check the execution state, or alternatively implement a reasonable timeout.

 

 

Executing workflows on external renderfarms

 

In principle (without a renderfarm) all nodes in a workflow are executed in serial manner, so no node will start until the previous has finished (and this is not different for pythin nodes). The same happens with the workflows in the queue, no parallel execution is possible within Workflows (or not without opening several Workflows applications at the same time).

The main reason for this is because most complex nodes (decoders, transcoders, display filters...) will use the GPU of the system very intensively, and overloading it can cause much slower global performance and stability issues (Mistika Workflows can not control multiple GPUs directly)

However, at the date of this document it is being integrated with 3rd party renderfarm management products such as Smedge & Deadline via connectivity plugins. This subject is out of the scope of this introduction, so we will only mention briefly:

By using multiple render nodes with those products it is possible to scale up the productivity up to any performance target. Two types of renderfarm buttons are provided:

- Renderfarm icon on renderfarm capable nodes: Activate it to render that single node in parallel (for example, a single transcoding node can be split into multiple render jobs, thus using many computers at the same time by rendering a packet of frames on each computer)

- Renderfarm icon on the queue panel: To submit each workflow as a separate render job, so that each whole workflow is sent to a different render node.

Logically each render node requires to have Mistika Workflows pre-installed before using this feature (the render manager just communicates with them). And a 3rd party render manager (Uberware Smedge or Deadline, which you can choose in the File->Preferences->Renderfarm menu) needs to be already running, as they are responsible for executing the jobs on each render node.

An introductory video using Deadline is available here.

#    if self.login=="":

    wf.giveBackAffinity()                     # this function will be needed in versions >= 10.4.0 to give the management of the created workflow back to the system
    return True

 

To create the node with the code above you can paste the node in the script editor, and using Right Mouse Button->Save In ibrary,

and call it "myExample". that will create a myExample.py in the SGOAppData/shared/workflowsLibrary directory and a myexample.xsd in the SGOAppData/shared/config/schemas directory.

 

the myexample.xsd needs to be edited in order to add the properties defined in the python script.

here you have an example of a myexample.xsd file with the properties added
