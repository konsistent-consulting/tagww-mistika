Standard properties

General properties

 

The properties in this section are available to multiple nodes, and this section is to have them documented in just one place. Nodes using them will just show the corresponding links to the property documentation here.

 

Object Name: Object Name is the name that identifies the node in your workflow as it appears in the top bar of the node block. By default it takes the generic name assigned to it in the list of nodes plus an instance number when appearing multiple times, but you can modify it to help understand the workflow scheme.

 

Color: Each node has a specific color assigned by default. The value of this property can be modified to customize its appearance. This can be useful to achieve more personalized and understandable workflow schemes for the user.

 

Lock:  This property will protect the node and all its properties from involuntary modifications, as no changes will be allowed until explicitly deactivating its Lock property again. It also provides a lock icon on top of the node for easy access. It is available in most of the nodes, except python nodes and some nodes not having any properties to protect.

 

Bypass: Enable or disable the bypass. When it is enabled the node won't be processed, but it will pass the inputs to the outputs. This function is available for most of the nodes but not all (this is a not a Workflows generic function,  each node is still responsible to select which inputs are passed to which outputs..). For example python nodes do not have it by default (see the mail node for an implementation example), and neither some switching or classifier nodes which purpose is precisely to divert traffic..

 

File Path (Directory, Name), Dst. Folder:

 

For the case of task nodes and output nodes these properties will define destination file paths.  

 

For the case of Input nodes, they will store the location of the source files.

 

A CRC property is also  provided for Input nodes, which will load the CRC of the file if a checksum sidecar file is also present (a checksum file is a companion file in the same folder with the same name and having a supported checksum extension, such as .md4 .md5 or .sha1). If present, it can be used by the Checksum node in order to validate the file integrity.

 

In general these properties can be set in one of these ways:

 

- By typing the path

- By pasting text

- By using drag & drop from a file browser

- By clicking the three-point "..." icon to open a file browser and select the desired file or folder.

 

In general, for simple cases we recommend the Drag & Drop action from a file browser to define the file path properties. It works with most nodes, it is the fastest method and it avoids typing mistakes.

 

By default, any path properties left empty will inherit the equivalent ones from the input files. For example, if you connect an input file to a transcoder task node without setting up any property then it will place the transcoded file in the same folder and also using the same filename,  thus only changing the extension.  Or if you only set the Directory then the file name will still come from the input file.

 

Also, a protection system is provided to avoid files overwritten with themselves, so if the input and output files have the same format an location an error will appear in the Validation Inspector. You can solve it either by changing the output Directory or the output Base Name.

 

When typing a path to a folder you should add and end slash "/" to the path. Please note that in many cases the path may not exist yet, so Workflows can't check if you have typed a file path or a folder path. Example:

 

If myFolder is a folder and not a regular file:

 

Correct path definition: /This/is/myFolder/

Ambigous path definition: /This/is/myFolder

 

Note for Python programmers:  The node connectors pass all the above metadata to other nodes, as what it goes trough the connectors are not just the file paths but a special objects including  all kind of useful metdaata called Universal paths, (the details of this mechanism are only necessary for programmers..  )

 

Also check that the name convention of the node matches the file paths that it is expected to manage. For example, if a node will receive files made of independent images with no frame numbers in their filenames you should use a name convention not containing the [frame] field in the Name Convention property (or just remove the [frame] tag  if it appeared by default...)

 

Many path style properties accept metadata token extraction / metadata token injection, For more details about name convention please read this chapter: Name convention

 

 

 

Universal path:

What travels trough the node connectors are not just "files" or "paths" but more complex data structures called "Universal Paths" (often abbreviated as "UP", or "url" in older versions ).  As a difference to a standard file path, a Mistika Universal Path can fully describe  complete enumerated sequences and other types of media packs made of multiple files,  including not only a file path (which is mandatory) but also all the metadata Name/Value pairs describing them such as frame ranges,  resolution, color space settings, timecodes, associated CRC checksums, server credentials to access the files  location, and any other metadata values embedded in the Universal Path by the previous nodes in the workflow.  

 

In this way no metadata is lost across the whole workflow  (each node can inject as much metadata as needed to the Universal Paths to be sent trough the output connectors, so all the previous metadata can be accessed by further nodes even if not contained by the passed files themselves ).

 

Most of the metadata being passed between nodes can be visualized with the pyPrintInputs node, which will dump it into the Script Editor panel. To better understand the Universal Paths data structures it is highly recommended to make some example workflows and then connect the pyPrintInputs node to see what is really going trough the connectors.

Transcoding standard properties

 

Note: An introduction to transcoding nodes is available in this video: video tutorial using transcoding nodes

 

Note: Not all the properties below are available to all the transcoding nodes.

 

Use AV audio: If active, the audio tracks will be taken from the embedded audio of the source movie files. If deactivated, the Ext.audio input connector will be used instead (multiple Ext.audio inputs can be created by doing drag & drop from other audio sources, but note that not all codecs support it).

 

Enable Frame Range (First frame, Duration): If enabled it will show a First frame and Duration properties to define which part of the input clips will be rendered (And passed to the next nodes).

 

Resolution: Horizontal and vertical resolution of the output images. You can select the most common resolution presets (HD, UHD, etc). The "Same as input" option keeps the input media resolution. "Custom" allows to set a Custom number  of pixels for vertical and horizontal. Two new fields are displayed:

 

Resolution X: Horizontal resolution of the output images. If you put a value of 0 (zero) it will appear as "From input", which is the default, and it means to use the same resolution as the input file.

 

Please note that some image formats require an even number of pixels, mainly because some color components may need to be shared between adjacent pixels (YUV22, YUV420, ...).

 

Resolution Y: Vertical resolution of the output images. If you put a value of 0 (zero) it will appear as "From input", which is the default, and it means to use the same resolution as the input file.

 

Please note that some image formats require an even number of lines (some color components may need to be shared between adjacent lines ( YUV420...).

 

Warning: Most formats have their own resolution limits, and error will appear at execution time if exceeded.  Few formats have unlimited resolution (EXR,  Mistika js are only limited by hardware capabilities).

 

Pixel Aspect Ratio: Changes the pixelAspectRatio., that is, the shape of individual pixels: Most modern imaging formats (HD, UHD...) use square pixels 1:1, but old television formats such as PAL, NTSC, and some film other formats use rectangular pixels with other aspect ratios. This property permits to change the pixelAspectRatio when rendering new media. The image resolution will be changed at render time, compressing the image in one direction (thus reducing the number of pixels) in order to match the new pixel aspect ratio.

 

Scale Filter: Defines which scaleFilter to use (rescaling algorithm) when changing image resolution at render time, both for scale up and scale down. This filter is applied when specifying a different resolution than the input images, and the possibilities are, from lower quality (but fast) to high quality (but slower):

 

Nearest

Bilinear

Bicubic

7 taps

15 taps

 

Note: 7 taps and 15 taps are based on complex Fourier transformations, and they are recommended for the highest quality (they are not based on  AI or any math tricks that could create artifacts, instead these modes will simulate an optical scaling.  They are slower (but still real-time with modern hardware) and in most cases provide the highest quality.

 

Fit: Indicates how to rescale the input images when their aspect ratio does not match the specified output resolution:

 

Disabled: It will pass the image pixels 1:1 with no scaling, no matter if part of the original image gets cropped in the process (center cropped in that case) or if it part of the new image is left black (black letterbox)

 

Fit: It will proportionally scale the image until it fits in the new resolution, but not cropping any part of the original image. (black letterbox is introduced as necessary).

 

Fill: It will proportionally scale the image until it fills all the pixels of the new resolution, thus cropping part of the original image if necessary.

 

Anamorphic: It will apply a non proportional scaling of the image, stretching or shrinking it by a different factor on each axis as necessary in order to match the defined resolution

 

Framerate: Frames per second value for the clip metadata. This is a metadata field that will tell the applications the FPS speed of the clip. Please note that changing it will not make any change to the media tracks themselves, only its metadata. Therefore, it can only be used to properly describe the content, not to do re-timing or frame rate conversions.

 

Interlaced: Indicates if the render or encoding process needs to be processed by frames (interlaced=Yes), by fields (interlaced=No) or keep as same as the input (Interlaces=From Input).

 

Overwrite TC: It allows to overwrite the TC. A new field called "New Timecode" is displayed when Overwrite TC is activated. A new TC can be added by using it.

 

First Frame From: When rendering to enumerated sequences it defines the first frame number to be used in their file names.

 

From Input: Same frame number as the source frame being transcoded.

 

From TC:  Set the frame number based on the time code of the source clip and its frame rate.  Logically this only works if the image format contains time code information, which should not be taken for granted.

 

Custom: A new property First Frame Nmber will appear, where the user can define this number.

 

Drop Frame:  Selects the type of Timecode (dropframe or non-dropframe). "From input" keeps the input media format (dropframe or non-dropframe) and "auto" function uses the input framerate to decide between  dropframe or non-dropframe based on the most common standards.

 

Audio Codec: Movie format nodes permit to encode embedded audio channels inside a movie file (AudioInMovie) or to produce external sidecar audio files. Meanwhile enumerated image formats can only encode it to external files (Wav16/24/32, Aiff 16bit, w64 32bit, caf 32bit, rf64 32bit). Please note that some audio formats are limited in size by their own specifications.

 

Many transcoder nodes (but not all) support multitrack audio, by connecting multiple audio files to the input side.

 

Note that old audio codecs such as wav are very compatible with other applications but do not support very long file sizes. If you need to encode long audio clips use more modern formats such as w64.

 

Render Farm: Enable or disable the external render farm selected (render managers from 3rd parties, such as Uberware Smedge and AWS Deadline). When it is selected, this node will send an .rnd to the selected render farm to process it externally.

 

The render farm type can be selected in File/Preferences/renderFarm

 

3rd party render managers are documented in their own manuals and not here.

 

Metadata properties

 

Here you can find a tutorial about Metadata Management in Mistika Workflows:  

https://youtu.be/2WC_hGpKkdw

 

Export Metadata (None, CSV, Xmp): It will create a .csv or .xmp text file in the destination folder of the current node, containing all the metadata obtained from the incoming media files, and also any additional metadata that was obtained, defined, or modified by the previous nodes.  

 

The exact metadata fields will depend on the capabilities of each previous node, some nodes can extract a lot of embedded metadata from their media files, while other nodes are more limited (or the additional metadata may not even exist in the original codec).

 

Note that a convenient method to control the metadata values is by using the Edit Metadata node (before the current node), which can modify any previous metadata or even create additional custom metadata pairs by using its Custom section.

 

Tape Name: The Tape Name is a special metadata field with a lot of implications for conforming media in other applications. It defines the principal name of the clip. The user can choose which one of the possible values will be embedded in the new media files as the new Tape Name:

 

Source Tape Name: The TapeName metadata field that comes  embedded inside internal headers of source media files (if any). It is the "name" field that you will see in the metadata properties when you import a media file into Mistika Workflows. but it does not need to match the file name at all (although it happens quite often)

 

Source File Name: The file name of the original media file.

 

Render File Name: The new file name that will be created for the rendered File, according to the current name convention.

 

Export CDL: When CDL metadata is passed to a node having this property active, it will create a sidecar .cc file (color primaries information) alongside with the rendered files, according to the ASC CDL standard.

 

Example:

 

Let's say that you import a sequence in this style:

 

Clipname.1000.dpx

Clipname.1001.dpx

Clipname.1002.dpx

...  

 

which is then passed to a Conform Filter node that is given an EDL containing CDL information. Then if you activate Export CDL in a posterior EXR transcoder node (for example) it will create a file named Clipname.exr.cc containing the CDL metadata, on the same folder as the destination media. So you will end with content like this in the destination folder:

 

Clipname.1000.exr

Clipname.1001.exr

Clipname.1002.exr

Clipname.exr.cc

 

It also works the other way: If you import said exr sequence and a Clipname.exr.cc file is found in the same folder then its CDL metadata will be available on posterior nodes, for example to be used by Export CDL on posterior transcoder nodes.

 

 

Advanced image processing properties

 

Lut3D: Apply a 3DLUT. Supported LUT3D formats are .cube, .itx and other formats following compatible industry standards.

 

Gamma, Gamut: A drop-down menu allows you to indicate the color space that best describes the images. You can select "Undefined" (in some nodes or "From input in another nodes) to keep default settings coming with the file or to select any other alternative from the list.  According to the selected color space, the corresponding values of Gamma and Gamut will be displayed.

 

In the case that this frame or sequence is going to be transcoded, it is important that these parameters are correctly identified and interpreted. In the case of input nodes, what you select here is just the color space of the original extraction, and this metadata information is then passed to task node transcoders which have the option to convert to other color spaces (using the Unicolor property).  But logically this conversion will only work properly if the original color space is accurately defined in the previous nodes.

 

The drop-down menu allows to select the possible values, but the default behavior is to simply pass through the available metadata settings coming with the original files (if they are defined). But sometimes they are undefined or unsupported by Mistika Workflows decoders, so you have these properties to declare them explicitly. If you see unexpected colors in the final images then you can use the Gamma and Gamut properties to declare specific values at import time.

 

In some cases both parameters can have limited options depending on the nature of the node.

 

Some  input nodes like Watcher and Folder  are color agnostic because they need to permit to import heterogeneous files, so they do not permit to set these properties. For these cases you can use the tagColorSpace task node. You will also need to use it for image formats coming wihout this information (or if Mistika Workflows can't find it)

 

Here you have a video tutorial about Color management in Mistika Workflows

 

 

Unicolor (gamma, gamut): Default: ---Same as Input--- (no changes).  This permits to apply a color space transformation (Output color space). This property permits to define the color space to be encoded. Choosing a particular color space from the list will select the destination Gamma curve and Gamut standard corresponding to it (For Mistika Boutique and Ultima users, this is the output field in ACES ODT FX) . Alternatively, it is also possible to select the option ---Custom Standard--- which will show two extra properties to select the gamma curve and gamut standard to be used  (For Mistika Boutique and Ultima users, this is the output field in Unicolor FX).

 

IMPORTANT NOTE: For the color spaces conversion to work, the workflow will also need to know the original color space of the input clip which has to specified in the Color Space properties of the input clip. In some cases the input clip node will set correct values for these properties by default, but in other cases it has to be set by the user. See gamma, gamut above for more details.

 

 

Display Filter: This is one of the most powerful features of Mistika technology. It permits to load a "Mistika Display Filter" to be applied to the images. The display filters are Mistika effect stacks created in other Mistika applications like Mistika Ultima and Mistika Boutique, which provide a complete library of effects. You can use display filters to apply custom color filters, watermarks, noise reduction, overlays, vfx, etc. You can watch a video tutorial here

 

Display filters are .dfilt files stored in the SGO AppData/shared/display_filters folder.  They have the same effect as if you apply them in Mistika Ultima & Boutique as display filters of "render" type.

 

A display filter is not limited to use the image clips passed to a node, but it can also reference its own image clips.  If a display filter was created using several image layers the Set DisplayFilter files node permits to substitute those images at execution time, just before using the display filter in another node.

 

If the display filter contains a Mistika multilayer effect (such as Mistika Comp3D) and the node is left in "From Input" state then the files coming from the input connectors will be used. This permits to apply complex composites without knowing the file paths in advance.

 

EXAMPLE:  The following example explains how to apply a logo overlay to any incoming media file, by using a display filter:

 

- In Mistika Boutique use a Comp3D effect to compose the logo over an image  clip (by now use any clip and logo you want, we will change it later...), add any other effects you want and create a display filter with the result. Test that it works as expected in Boutique.  Just make sure the display filter is saved in the shared/display_filters folder as usual.  

 

- Now open Mistika Workflows and connect a Watcher node to a transcoder task node of the desired type (for example a ProRes task node), and leave its file path as "From Input"

 

- In the transcoder node -> display filter property select the display filter that you created in Mistika Boutique.

 

- Add the workflow to the queue. From this point every time that a media file appears in the Watcher folder the logo will be composed as an overlay on top of the input media as part of the transcoder execution (this method is optimal to maintain the best quality, as both the compositing and the transcoding will happen at the same time, thus avoiding two generations of compression). The files passed trough its input connector are automatically assigned as layer1 of the display filter (background of the composition)

 

If you also want to change the logo image then you can add an independent flow for this (an independent "island" in the same workflow) :

 

-  create a SetDisplayFilterFiles node, and select the same display filter as in the above exampe

 

-  connect a node containing the new logo to the SetDisplayFilterFiles node.  This node will modify the display filter at execution time, so that it will use the new file path (you could also redirect the result to a new display filter file if you want, but it is not necessary for this example).

 

-  connect the rank connector of the SetDisplayFilterFiles node to the transcoder node (ProRes in the above example), to ensure the SetDisplayFilterFiles branch of the workflow is executed first. (do not connect the SetDisplayFilterFiles output to the transcoder node input, only the rank link must be connected. The rank link will just establish the execution order (modifying the display filter before using it in the transcoder node)

 

VirtualSlate, Virtual Slate Duration: Superimpose clip metadata information as an image overlay, for the specified duration (a number of start frames or permanent).

 

 

Folder standard properties

 

Recursive / Include Subfolders: The recursive mode allows the connection of all the files that are included in any subfolders of the selected path.

 

Include: You can determine the type of files with which Watcher will react according to their name or extension. In case of writing *. * it will be activated with any name and file extension.

 

Exclude: You can assign the types of files that Watcher will ignore, for example * .mov, * .avi.

 

Note: Include and Exclude accept a comma separated list, and also support wildcards "*" as most operating systems do.

 

Filter Mode: How to parse the include / exclude properties for matching filenames.

 

Text: Characters are interpreted as they are. Symbols like [ or # do not have any special meaning.

 

Glob: Equivalent to the Glob command used on file browsers of the operating system. For example you can use ? and * for matching.  Something like A*.r3d will match all r3d files starting with an "A".

 

Regular Expressions: Accepts regular expressions (regexp).

 

IncludeHidden: Process hidden files

 

IncludeSystem: Process system files

 

Alphabetical Order: Send files trough the output connector in alphabetical order.

 

Add Root: Add Root: By default, nodes processing folder files will get any file landing in sub-directory recursively, but the directory hierarchy is ignored for the output connector. For example, if you are transcoding the contents of a Folder or a Watcher folder by connecting them to a ProRes node, by default all the rendered files will be sent to the same folder despite of the original  subfolder structures.

If you want to change the default behavior to maintain the original hierarchy then activate the Add Root property.

 

Python standard properties

 

Code: Source code to be executed by the node, written in Python programming language. Some nodes are implemented in python, in which case the source code is made available in the code property for the interest of advanced users.  For more details, the Mistika Workflows tools related with python are described in the Python scripting tools section

 

For more details, read the Python scripting tools chapter

 

 

Schema name: Path to an .xsd file to display properties created in a certain node.  The .xsd file is an editable file in .xsd format.

A typical use is to define how to display the new properties created by Python code in the properties panel of the node for interaction with the final user.  The python code can define a lot of data structures with cryptic names and the purpose of the schema file is to only display those properties intended for user interaction, also using friendly labels for them.  

 

Example: A copy node with an added property visible in the GUI.

 

This node just copies the input to the path defined in the property dstFile and sends it to the output for the next node.

 

Code:

 

from shutil import copyfile

 

def init(self):     # initialization function to create in/out connectors and properties (attributes)

self.addConnector("in",1,1)        # 1=input. 1=mandatory

self.addConnector("out",2,0)        # 2=output. 0=optional

self.addProperty('dstFile')

return True                # successful initialization

 

def isReady(self):  # check if the node can be processed. in this example, dstFile needs to be defined

ifself.dstFile!='':

 return self.critical('myErrorID','dstFile can not be empty')

return True

 

def process(self):    #copy in file to dst File and set output connector)

input=self.firstConnector#("in")

output=self.firstConnectorByName("out")

copyfile(input.url(),self.dstFile)

output.setUrl(self.dstFile)

return True        # performed ok (no error management in this example)

 

 

Schema:

 

In order to make it work correctly, you will need to create an schema file for the node  properties. it should be in "SGO AppData/shared/config/schemas" and it should be something like this: (myCopy.xsd)

 

 

Note: This example is just a copy of python.xsd but adding the “dstFile” property:

 

 

<xsd:schema xmlns:sgo="http://www.sgo.es/2017/sgo.xsd"

          xmlns:xsd="http://www.w3.org/2001/XMLSchema">

<xsd:element sgo:categoryType="categoryCollapsible" name="Python" sgo:visibility="0" sgo:sortOrder="0">

  <xsd:complexType>

      <xsd:sequence>

          <xsd:element name="objectName" type="xsd:string" sgo:visibility="0" sgo:sortOrder="0"/>

          <xsd:element name="color" type="color" sgo:visibility="0" sgo:sortOrder="1"/>

          <xsd:element name="dstFile" type="universalPath" sgo:visibility="0" sgo:sortOrder="2" sgo:properties="pathNameFilters('All Files (*.*)')"/>

          <xsd:element name="schemaName" type="xsd:anyURI" sgo:visibility="0" sgo:sortOrder="3" sgo:properties="pathNameFilters('xsd Files (*.xsd)')"/>

          <xsd:element name="code" type="textEditor" sgo:visibility="0" sgo:sortOrder="4"/>

      </xsd:sequence>

  </xsd:complexType>

</xsd:element>

</xsd:schema>

 

 

If you want to remove the code and some schema details (once the node is finished and well tested) you can also remove default lines, in the next example we have removed the source code and the scheme file, to keep it out of sight for normal users:

 

 

<xsd:schema xmlns:sgo="http://www.sgo.es/2017/sgo.xsd"

          xmlns:xsd="http://www.w3.org/2001/XMLSchema">
